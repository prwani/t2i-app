Build a Microsoft Foundry image asset workflow with GPT and MAI image models

Generative image models are most useful to product, marketing, and design teams when they are part of a repeatable workflow: prompt, generate, compare, evaluate, refine, and package the best assets for the channel where they will be used. This article walks through a practical reference app that brings that loop together with Microsoft Foundry image models, Azure AI Vision, a Python SDK, and a deployable web experience.

Who this is for: developers, and the product, marketing, and design teams they support, who need to produce on-brand image assets at scale and want an objective, repeatable way to choose the best candidate — not just a one-off prompt. In about ten minutes locally, you can generate several variations of an asset, score each one for prompt adherence and visual quality, and rank them automatically before a human gives final sign-off.

What you will build: a local and cloud-ready image asset workflow that can generate images with GPT image and MAI (Microsoft AI) image deployments, evaluate prompt adherence and visual quality, compare candidates, and rank generated variations.
Disclaimers: The sample code in this article is provided “as is” for illustration and learning, without warranties of any kind; review and test it before any production use. The deployment names shown are examples — use the exact model names available in your Azure region. Why this pattern

A single text-to-image call can create a useful draft, but production asset creation usually needs more structure:

  • Model choice: use MAI image deployments, including MAI-Image-2.5 and MAI-Image-2.5-Flash, for text-to-image generation and single-image editing workflows. Keep GPT image available for editing and for multi-image composition.
  • Scenario guidance: apply repeatable prompt patterns for brand templates, text rendering, aspect-ratio adaptation, multi-turn refinement, and product placement.
  • Objective review: evaluate every candidate with the same prompt-adherence and quality rubric instead of relying only on manual inspection.
  • Deployment readiness: keep local experimentation and Azure Container Apps deployment on the same configuration model.

Solution architecture

The repository is organized around a reusable Python SDK named t2i_core (“text-to-image core”). The SDK owns provider construction, model calls, scenario orchestration, evaluation, settings, and utility functions. The web UI and agent skills call into the SDK rather than duplicating model-specific logic.

The app keeps t2i_core as the backend domain layer, exposes it through a FastAPI service, and uses a Next.js front end. Evaluation and compare workflow

Evaluation is built as layers so teams can choose the right cost and latency profile for each step. The goal is not to replace human creative review; it is to make the first-pass comparison repeatable, explainable, and consistent across generated candidates.

There are three scoring layers, in increasing order of cost and depth: the embedding layer for fast semantic screening, the rubric layer for attribute-level prompt adherence, and the judge layer, where a vision model rates visual quality, spatial coherence, style, aesthetics, and text legibility. The composite score and the decision then roll those layers up into a single rank and a recommended next action. The Layer column below shows which metric belongs to which.

Layer Metric What it measures How to read it Example
Embedding Embedding similarity Broad semantic alignment between the prompt text and the image, using Azure AI Vision multimodal vectors. A cosine similarity score. It is useful for fast screening, but it is not a percentage and should be calibrated for your asset category. Prompt: "a red backpack on a wooden desk." An image containing a red backpack on a desk should score higher than a beach landscape, even before checking detailed attributes.
Rubric Rubric score Prompt adherence at the attribute level. The app decomposes the prompt into concrete checks and verifies whether each one is present. A 0-1 score where each "yes" counts as 1, each "partial" counts as 0.5, and each "no" counts as 0. If the prompt asks for "a blue ceramic mug, steam rising, on a marble counter" and the image has the mug and counter but no visible steam, the rubric might mark two attributes as yes and one as no, resulting in 0.67.
Judge Visual quality Whether the image is sharp, clean, artifact-free, and suitable as a production asset. A 1-5 judge score, where 5 means strong production quality and 1 means severe defects. A product photo with crisp edges and no visible distortions may receive 5; a warped product label or noisy background may receive 2.
Judge Spatial coherence Whether objects, perspective, lighting, hands, reflections, shadows, and composition are physically plausible. A 1-5 judge score focused on scene consistency rather than prompt matching. A handbag correctly resting on a table with consistent shadowing scores higher than one floating above the table or casting an impossible shadow.
Judge Style fidelity Whether the image follows the requested visual style, such as photorealistic product photography, flat illustration, luxury editorial, or social ad creative. A 1-5 judge score. This is especially useful when comparing variants that all contain the requested objects but differ in brand fit. For "minimal premium skincare ad on a warm beige background," a cluttered neon cyberpunk result may have good objects but low style fidelity.
Judge Aesthetic appeal Overall creative polish: composition, balance, color harmony, framing, and marketability. A 1-5 judge score that helps rank multiple acceptable candidates. Two images may both satisfy the prompt, but the one with stronger framing, cleaner negative space, and better color contrast should rank higher.
Judge Text legibility Whether generated text is readable and accurate when the prompt asks for visible text, packaging copy, signage, or labels. An optional 1-5 judge score. It is included only when text is relevant to the prompt. For "a poster that says SUMMER SALE," a clear and correctly spelled headline scores high; misspelled or melted lettering scores low.
Composite Composite score A normalized summary score across the selected layers. A 0-1 weighted score. If you skip a layer, the active weights are renormalized instead of treating the skipped layer as zero. If only embedding and rubric are selected, the composite ranks candidates using those two signals. If the judge layer is added, quality and aesthetics also influence the final order.
Decision Decision A workflow recommendation: accept, review, or regenerate. The app compares layer scores and the composite score with configurable thresholds. A candidate with strong visual quality but poor rubric adherence may be marked regenerate because it looks good but missed the requested product, color, or scene requirement.

A practical pattern is to use embedding similarity for inexpensive early filtering, add rubric scoring when prompt adherence matters, and add the judge layer when visual polish, brand fit, or text legibility affects the final choice. For example, a marketing team generating five product-ad variations can rank all five by composite score, then inspect the per-metric explanations to understand whether the winner was selected because it matched the prompt, looked more polished, or both. 

report = await EvaluationPipeline(settings).evaluate(
    image_bytes,
    prompt,
    layers=["embedding", "rubric", "judge"],
)

In the UI, creators can generate candidates, evaluate generated results, compare uploaded images against a shared prompt, or run a batch rank workflow that generates variations and orders them by score.

Worked example: how the rubric layer scores one image

The rubric layer is the easiest to reason about because every step is explicit. Rather than asking a model “is this a good image?”, the app turns the prompt into a checklist of concrete, verifiable attributes, then asks a vision model to mark each one. The example below walks through a single image end to end.

Step 1 — Start with the prompt. The creator submits one generation prompt: 

A blue ceramic coffee mug on a white marble counter, hot steam
rising from the top, soft morning light from the left, shallow
depth of field, photorealistic product photography

Step 2 — The decomposer turns the prompt into rubric questions. The EVAL_DECOMPOSER_DEPLOYMENT model breaks the prompt into independent yes/no checks. Each check targets one attribute so the score is explainable rather than a single opaque number:

  • Is the main subject a coffee mug?
  • Is the mug blue?
  • Is the mug made of ceramic?
  • Is the mug resting on a white marble counter?
  • Is hot steam visibly rising from the mug?
  • Does the lighting read as soft morning light coming from the left?
  • Is there a shallow depth of field (blurred background)?
  • Is the overall style photorealistic product photography?
Image placeholder: the generated candidate being scored — a blue ceramic mug on a marble counter, side-lit, blurred background, with no visible steam.
Alt text to add: “A blue ceramic coffee mug on a white marble counter, soft light from the left and a blurred background, generated by a Microsoft Foundry image model.”

Step 3 — The rubric model grades the image against each question. The EVAL_RUBRIC_DEPLOYMENT model inspects the generated image and returns a verdict and a short reason for every attribute. Scoring uses yes = 1, partial = 0.5, no = 0:

Rubric question Verdict Score Model’s reason
Is the main subject a coffee mug? Yes 1.0 A single mug is clearly the focal subject.
Is the mug blue? Yes 1.0 The mug is a solid medium blue.
Is the mug made of ceramic? Yes 1.0 Matte glazed surface consistent with ceramic.
Is the mug on a white marble counter? Yes 1.0 Light surface with grey veining reads as marble.
Is hot steam visibly rising? No 0.0 No steam or vapor is present above the mug.
Soft morning light from the left? Yes 1.0 Highlights and shadows indicate a soft left-side key light.
Shallow depth of field? Partial 0.5 Background is softened but not strongly blurred.
Photorealistic product photography style? Yes 1.0 Lighting, focus, and texture read as a real product photo.

Step 4 — Roll the verdicts into a score and a decision. Summing the verdicts gives 6.5 out of 8 attributes, for a rubric score of 0.81. The judge layer separately rates visual quality and the other 1-5 dimensions, and the composite score combines the active layers. Because the one missing attribute — steam — is easy to name, the workflow can surface a precise recommendation: regenerate with stronger emphasis on visible rising steam, rather than a vague “try again.” That traceability from prompt → questions → per-attribute verdicts → score is what makes the rubric layer useful for comparing several candidates objectively.

  • Python 3.11.
  • Azure CLI for local Microsoft Entra authentication.
  • Access to Microsoft Foundry or Azure OpenAI image deployments.

3. Generate channel-ready assets

The app supports API-supported generation sizes such as 1024x1024, 1536x1024, and 1024x1536. Format helpers can map common channel targets such as instagram_square, instagram_story, linkedin_banner, desktop_hero, and mobile_story to supported model sizes.

A practical workflow for a campaign asset is:

  1. Select a generation-only scenario and start with MAI-Image-2e for efficient exploration, MAI-Image-2.5-Flash for faster production runs, or MAI-Image-2.5 when maximum fidelity is the priority.
  2. Use prompt improvement to add scene structure, subject details, lighting, constraints, and brand context.
  3. Generate several candidates.
  4. Use the scenario-filtered model selector: single-image edit workflows can use GPT image, MAI-Image-2.5-Flash, or MAI-Image-2.5, while multi-image composition remains GPT image only.
  5. Evaluate and compare candidates before exporting.

4. Use the SDK from scripts or developer notebooks

The SDK is intentionally notebook-friendly. That makes it useful for prompt recipe development, evaluator calibration, and repeatable experiments before a workflow is promoted into the UI. 

from t2i_core import EvaluationPipeline, GPTImageProvider, MAIImageProvider, Settings

settings = Settings()

gpt_provider = GPTImageProvider(settings)
mai_provider = MAIImageProvider(settings, deployment=settings.mai_image_efficient_deployment)
mai25_flash_provider = MAIImageProvider(settings, deployment=settings.mai_image_25_flash_deployment)
mai25_provider = MAIImageProvider(settings, deployment=settings.mai_image_25_deployment)

Recommended notebook sections include:

  • Prompt recipe: prompt, negative constraints, target channel, and model choice.
  • Generation run: parameters, image outputs, and metadata.
  • Evaluation report: selected layers, thresholds, scores, and notes.
  • Decision log: why one candidate was selected for refinement or publication.

5. Validate before deployment

Run the SDK tests to catch regressions in providers, settings, and evaluation helpers: 

pytest packages/t2i_core/tests

For live validation, generate a small number of low- or medium-quality images, run embedding-only evaluation first, and then add rubric or judge layers when you need a deeper review.

Production considerations

  • Storage: the current API stores generated images on local writable container storage. For multi-user production, add Azure Blob Storage for durable asset libraries and audit trails.
  • Identity: prefer managed identity in Azure and DefaultAzureCredential locally. Avoid API keys in source code.
  • Cost controls: default to embedding-only or generation-only checks during ideation, then run rubric and judge layers for shortlisted candidates.
  • Human review: keep a final human approval step for brand, accessibility, legal, and regional requirements.
  • Frontend evolution: extend the Next.js app with saved projects, collaborative review, richer galleries, or integration with enterprise design systems.
  • Backend evolution: add durable jobs, queue-based generation, signed asset downloads, and central policy enforcement.

To stop local execution, end the FastAPI and Next.js processes in your terminals. For cloud resources, remove only the resources created for this walkthrough if they are no longer needed:
