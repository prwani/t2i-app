Build a Microsoft Foundry image asset workflow with GPT and MAI image models

Generative image models are most useful to product, marketing, and design teams when they are part of a repeatable workflow: prompt, generate, compare, evaluate, refine, and package the best assets for the channel where they will be used. This article walks through a practical reference app that brings that loop together with Microsoft Foundry image models, Azure AI Vision, a Python SDK, and a deployable web experience.

What you will build: a local and cloud-ready image asset workflow that can generate images with GPT-Image and MAI-Image deployments, evaluate prompt adherence and visual quality, compare candidates, and rank generated variations.

Why this pattern

A single text-to-image call can create a useful draft, but production asset creation usually needs more structure:

  • Model choice: use MAI image deployments, including MAI-Image-2.5 and MAI-Image-2.5-Flash, for text-to-image generation and single-image editing workflows. Keep GPT image available for editing and for multi-image composition.
  • Scenario guidance: apply repeatable prompt patterns for brand templates, text rendering, aspect-ratio adaptation, multi-turn refinement, and product placement.
  • Objective review: evaluate every candidate with the same prompt-adherence and quality rubric instead of relying only on manual inspection.
  • Deployment readiness: keep local experimentation and Azure Container Apps deployment on the same configuration model.

Solution architecture

The repository is organized around a reusable Python SDK named t2i_core. The SDK owns provider construction, model calls, scenario orchestration, evaluation, settings, and utility functions. The web UI and agent skills call into the SDK rather than duplicating model-specific logic.

The current app keeps t2i_core as the backend domain layer, exposes it through a FastAPI service, and uses a Next.js front end for richer multi-user experiences, saved asset libraries, approval workflows, and integration with design systems.

Image generation capabilities

Scenario Recommended provider Typical use
Text-to-image generation GPT image or MAI image Create net-new visuals from a prompt.
Brand template GPT image or MAI image Apply brand colors, tone, product context, and layout constraints.
Text rendering MAI-Image-2.5 or GPT image Generate assets where readable text matters.
Aspect-ratio adaptation GPT image or MAI image Create channel-ready sizes such as square, story, banner, hero, or mobile formats.
Inpainting, product placement, refinement GPT image, MAI-Image-2.5-Flash, or MAI-Image-2.5 Use one source image, optional masks, or previous outputs to create controlled edits.
Multi-image composition GPT image Combine two or more source images into a new controlled asset.
Batch variations and ranking GPT image plus evaluation Generate multiple options and sort them with a normalized composite score.

Evaluation and compare workflow

Evaluation is built as layers so teams can choose the right cost and latency profile for each step:

  • Embedding: uses Azure AI Vision vectors for broad semantic alignment between the prompt and image.
  • Rubric: decomposes the prompt into visual attributes and verifies each attribute.
  • Judge: reviews visual quality, coherence, style, aesthetics, and text legibility when relevant.
  • Pipeline: combines selected layers into a normalized composite score. Skipped layers are not treated as zeroes.
report = await EvaluationPipeline(settings).evaluate(
    image_bytes,
    prompt,
    layers=["embedding", "rubric", "judge"],
)

In the UI, creators can generate candidates, evaluate generated results, compare uploaded images against a shared prompt, or run a batch rank workflow that generates variations and orders them by score.

Prerequisites

  • Python 3.11.
  • Azure CLI for local Microsoft Entra authentication.
  • Access to Microsoft Foundry or Azure OpenAI image deployments.
  • Azure AI Vision endpoint for embedding-based evaluation.
  • Deployment names for GPT image, MAI image, MAI 2.5 image, and evaluation model deployments.
  • Optional: Azure Container Registry and Azure Container Apps for deployment.
  • For Azure deployment: an existing Microsoft Foundry resource and project, with deployment details already captured in .env.

1. Configure the local environment

Start with a Python environment and install the local SDK with API dependencies: 

uv venv --python 3.11 .venv
source .venv/bin/activate
uv pip install -e "packages/t2i_core[dev,api]"
cp .env.example .env

Update .env with your resource endpoints and deployment names. Use environment variable names rather than hard-coding values in notebooks, scripts, or UI code. 

FOUNDRY_PROJECT_ENDPOINT=https://<resource>.services.ai.azure.com/api/projects/<project>
AZURE_OPENAI_ENDPOINT=https://<resource>.openai.azure.com
AZURE_VISION_ENDPOINT=https://<resource>.cognitiveservices.azure.com
GPT_IMAGE_2_DEPLOYMENT=gpt-image-2
MAI_IMAGE_2_DEPLOYMENT=MAI-Image-2
MAI_IMAGE_2E_DEPLOYMENT=MAI-Image-2e
MAI_IMAGE_2_5_FLASH_DEPLOYMENT=MAI-Image-2.5-Flash
MAI_IMAGE_2_5_DEPLOYMENT=MAI-Image-2.5
EVAL_DECOMPOSER_DEPLOYMENT=o4-mini
EVAL_RUBRIC_DEPLOYMENT=o4-mini
EVAL_JUDGE_DEPLOYMENT=gpt-5.4

Authenticate locally with Microsoft Entra ID: 

az login

2. Run the web app locally

Launch the FastAPI backend: 

uvicorn api.main:app --reload --port 8000

In a second terminal, launch the Next.js frontend: 

cd web
npm install
npm run dev

3. Generate channel-ready assets

The app supports API-supported generation sizes such as 1024x1024, 1536x1024, and 1024x1536. Format helpers can map common channel targets such as instagram_square, instagram_story, linkedin_banner, desktop_hero, and mobile_story to supported model sizes.

A practical workflow for a campaign asset is:

  1. Select a generation-only scenario and start with MAI-Image-2e for efficient exploration, MAI-Image-2.5-Flash for faster production runs, or MAI-Image-2.5 when maximum fidelity is the priority.
  2. Use prompt improvement to add scene structure, subject details, lighting, constraints, and brand context.
  3. Generate several candidates.
  4. Use the scenario-filtered model selector: single-image edit workflows can use GPT image, MAI-Image-2.5-Flash, or MAI-Image-2.5, while multi-image composition remains GPT image only.
  5. Evaluate and compare candidates before exporting.

4. Use the SDK from scripts or developer notebooks

The SDK is intentionally notebook-friendly. That makes it useful for prompt recipe development, evaluator calibration, and repeatable experiments before a workflow is promoted into the UI. 

from t2i_core import EvaluationPipeline, GPTImageProvider, MAIImageProvider, Settings

settings = Settings()

gpt_provider = GPTImageProvider(settings)
mai_provider = MAIImageProvider(settings, deployment=settings.mai_image_efficient_deployment)
mai25_flash_provider = MAIImageProvider(settings, deployment=settings.mai_image_25_flash_deployment)
mai25_provider = MAIImageProvider(settings, deployment=settings.mai_image_25_deployment)

Recommended notebook sections include:

  • Prompt recipe: prompt, negative constraints, target channel, and model choice.
  • Generation run: parameters, image outputs, and metadata.
  • Evaluation report: selected layers, thresholds, scores, and notes.
  • Decision log: why one candidate was selected for refinement or publication.

5. Validate before deployment

Run the SDK tests to catch regressions in providers, settings, and evaluation helpers: 

pytest packages/t2i_core/tests

For live validation, generate a small number of low- or medium-quality images, run embedding-only evaluation first, and then add rubric or judge layers when you need a deeper review.

6. Deploy to Azure Container Apps

The repository includes a container path for Azure Container Apps. In the tested deployment, the Microsoft Foundry resource/project already existed, the local .env file held the project endpoint and deployment names, and the Container App used a user-assigned managed identity for model access. 

RG=<resource-group>
LOCATION=<azure-region>
ACR=<globally-unique-acr-name>
APP=t2i-app
ENV=t2i-env
LAW=t2i-law
MI=t2i-app-mi

Create the container registry, user-assigned managed identity, Log Analytics workspace, and Container Apps environment. Creating Log Analytics explicitly avoids ambiguous waits during automatic workspace creation: 

az acr create --name "$ACR" --resource-group "$RG" --location "$LOCATION" --sku Basic
az identity create --name "$MI" --resource-group "$RG" --location "$LOCATION"
az monitor log-analytics workspace create --resource-group "$RG" --workspace-name "$LAW" --location "$LOCATION"

LAW_ID=$(az monitor log-analytics workspace show \
  --resource-group "$RG" --workspace-name "$LAW" --query customerId -o tsv)
LAW_KEY=$(az monitor log-analytics workspace get-shared-keys \
  --resource-group "$RG" --workspace-name "$LAW" --query primarySharedKey -o tsv)

az containerapp env create \
  --name "$ENV" \
  --resource-group "$RG" \
  --location "$LOCATION" \
  --logs-workspace-id "$LAW_ID" \
  --logs-workspace-key "$LAW_KEY"

Grant the managed identity ACR pull access and model access. Assign Cognitive Services OpenAI User and Cognitive Services User on the Azure OpenAI or Azure AI Services resource, and also at the Microsoft Foundry project scope if your tenant requires project-level access: 

MI_PRINCIPAL=$(az identity show --name "$MI" --resource-group "$RG" --query principalId -o tsv)
ACR_ID=$(az acr show --name "$ACR" --resource-group "$RG" --query id -o tsv)
AI_ACCOUNT_ID=<cognitive-services-or-ai-services-account-resource-id>
FOUNDRY_PROJECT_ID=<microsoft-foundry-project-resource-id>

az role assignment create --assignee-object-id "$MI_PRINCIPAL" --assignee-principal-type ServicePrincipal --role AcrPull --scope "$ACR_ID"
az role assignment create --assignee-object-id "$MI_PRINCIPAL" --assignee-principal-type ServicePrincipal --role "Cognitive Services OpenAI User" --scope "$AI_ACCOUNT_ID"
az role assignment create --assignee-object-id "$MI_PRINCIPAL" --assignee-principal-type ServicePrincipal --role "Cognitive Services User" --scope "$AI_ACCOUNT_ID"
az role assignment create --assignee-object-id "$MI_PRINCIPAL" --assignee-principal-type ServicePrincipal --role "Cognitive Services OpenAI User" --scope "$FOUNDRY_PROJECT_ID"
az role assignment create --assignee-object-id "$MI_PRINCIPAL" --assignee-principal-type ServicePrincipal --role "Cognitive Services User" --scope "$FOUNDRY_PROJECT_ID"

Build with a unique image tag so Azure Container Apps creates a fresh revision instead of reusing a cached latest image: 

TAG=deploy-$(date +%Y%m%d%H%M%S)
az acr build --registry "$ACR" --image "t2i-app:$TAG" .

Reuse the local .env file for Container App environment variables and add AZURE_CLIENT_ID for the user-assigned identity: 

MI_ID=$(az identity show --name "$MI" --resource-group "$RG" --query id -o tsv)
MI_CLIENT_ID=$(az identity show --name "$MI" --resource-group "$RG" --query clientId -o tsv)

python - <<'PY' > /tmp/t2i-env-vars.txt
from pathlib import Path

keys = [
    "FOUNDRY_PROJECT_ENDPOINT",
    "AZURE_OPENAI_ENDPOINT",
    "AZURE_VISION_ENDPOINT",
    "MAI_IMAGE_2_DEPLOYMENT",
    "MAI_IMAGE_2E_DEPLOYMENT",
    "MAI_IMAGE_2_5_FLASH_DEPLOYMENT",
    "MAI_IMAGE_2_5_DEPLOYMENT",
    "GPT_IMAGE_2_DEPLOYMENT",
    "EVAL_DECOMPOSER_DEPLOYMENT",
    "EVAL_RUBRIC_DEPLOYMENT",
    "EVAL_JUDGE_DEPLOYMENT",
]

values = {}
for line in Path(".env").read_text(encoding="utf-8").splitlines():
    line = line.strip()
    if not line or line.startswith("#") or "=" not in line:
        continue
    key, value = line.split("=", 1)
    values[key.strip()] = value.strip().strip('"').strip("'")

values["AZURE_CLIENT_ID"] = "__AZURE_CLIENT_ID__"
for key in [*keys, "AZURE_CLIENT_ID"]:
    value = values.get(key)
    if value:
        print(f"{key}={value}")
PY

sed -i "s#__AZURE_CLIENT_ID__#$MI_CLIENT_ID#" /tmp/t2i-env-vars.txt

Create the Container App with external ingress, managed identity for ACR pull, and the same managed identity for model access: 

IMAGE="$ACR.azurecr.io/t2i-app:$TAG"

az containerapp create \
  --name t2i-app \
  --resource-group "$RG" \
  --environment "$ENV" \
  --image "$IMAGE" \
  --target-port 8000 \
  --ingress external \
  --registry-server "$ACR.azurecr.io" \
  --registry-identity "$MI_ID" \
  --user-assigned "$MI_ID" \
  --env-vars $(cat /tmp/t2i-env-vars.txt)

For an existing app, update the image and environment variables: 

az containerapp update \
  --name "$APP" \
  --resource-group "$RG" \
  --image "$IMAGE" \
  --revision-suffix "${TAG//deploy-/d}" \
  --set-env-vars $(cat /tmp/t2i-env-vars.txt)

After deployment, enable Microsoft Entra authentication at the Container Apps ingress layer and restrict access to the tenant, application, group, or app role appropriate for your deployment.

Use the FastAPI health endpoint and scenario catalog for probes: 

FQDN=$(az containerapp show --name "$APP" --resource-group "$RG" --query properties.configuration.ingress.fqdn -o tsv)
curl -fsS "https://$FQDN/health"
curl -fsS "https://$FQDN/api/scenarios"

To verify managed identity access to the configured models, run one generation request: 

curl -fsS --max-time 360 \
  -X POST "https://$FQDN/api/generations" \
  -H "Content-Type: application/json" \
  -d '{
    "scenario": "text-to-image",
    "model": "MAI-Image-2e",
    "prompt": "A simple blue circle icon on a white background, minimal vector style, no text",
    "size": "1024x1024",
    "quality": "low",
    "n": 1
  }'

Production considerations

  • Storage: the current API stores generated images on local writable container storage. For multi-user production, add Azure Blob Storage for durable asset libraries and audit trails.
  • Identity: prefer managed identity in Azure and DefaultAzureCredential locally. Avoid API keys in source code.
  • Cost controls: default to embedding-only or generation-only checks during ideation, then run rubric and judge layers for shortlisted candidates.
  • Human review: keep a final human approval step for brand, accessibility, legal, and regional requirements.
  • Frontend evolution: extend the Next.js app with saved projects, collaborative review, richer galleries, or integration with enterprise design systems.
  • Backend evolution: add durable jobs, queue-based generation, signed asset downloads, and central policy enforcement.

Cleanup

To stop local execution, end the FastAPI and Next.js processes in your terminals. For cloud resources, remove only the resources created for this walkthrough if they are no longer needed: 

az containerapp delete --name t2i-app --resource-group <resource-group>
az acr repository delete --name <acr-name> --image t2i-app:<tag>
